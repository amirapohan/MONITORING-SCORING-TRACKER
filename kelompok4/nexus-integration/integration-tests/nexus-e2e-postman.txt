BASE URL (API Gateway tunggal):
    http://map-sandbox.tailcbdd04.ts.net   ->  {{BASE}}

Catatan gateway dosen: prefix /auth /bidding /match /tracker /notify dipetakan
ke mount internal tiap service. PAKAI PERSIS path di bawah (jangan tambah
/api atau /api/v1 sendiri — gateway sudah menanganinya).
    /auth/X    -> svc-auth   /api/auth/X        (atau /internal/* utk service)
    /bidding/X -> svc-bidding/api/X
    /match/X   -> svc-match  /X
    /tracker/X -> svc-audit  /api/v1/X
    /notify/X  -> svc-notify /api/X

----------------------------------------------------------------------------
 SERVICE vs PERSONA — JANGAN TERTUKAR
----------------------------------------------------------------------------
 K1..K5 = SERVICE (microservice/tim), BUKAN orang:
   K1 Identity/SSO  · K2 Project Bidding · K3 Team Matching
   K4 Tracker (grup kita) · K5 Notification
 Persona (role user, semuanya disimpan di K1):
   - TALENT (mahasiswa/role "talent")  -> mengerjakan
   - CLIENT (mitra/employer/role "client") -> posting project, bayar, me-review
 Jadi "K4 buat milestone" = fitur milestone di service K4 yang dipakai CLIENT
 untuk seorang TALENT. Bukan "K4 = talent".

----------------------------------------------------------------------------
 ALUR DUNIA-NYATA yang diuji (project DULU, baru milestone)
----------------------------------------------------------------------------
 CLIENT buat PROJECT (K2) -> TALENT BID (K2) -> CLIENT ACCEPT bid = talent
 "awarded" (K2) -> CLIENT buat MILESTONE utk talent itu, tertaut projectId (K4)
 -> TALENT submit bukti + file (K4/MinIO) -> CLIENT review (K4 verifikasi ke K1)
 -> notifikasi (K5) di tiap langkah.

----------------------------------------------------------------------------
 PENTING — MODE VALIDASI (env VALIDATE_INTEGRATION di svc-audit)
----------------------------------------------------------------------------
 - OFF (default di deployment live): K4 menerima employerId/studentId/projectId
   apa adanya (kompatibel dgn data & test lama). Alur di bawah TETAP jalan.
 - ON (saat ingin menegakkan aturan real-world): K4 memvalidasi ke K1 & K2:
     * employerId harus user role "client" di K1
     * studentId  harus user role "talent" di K1
     * bila projectId diisi: project harus ADA di K2 DAN studentId harus
       sudah ACCEPTED (awarded) pada project itu.
 Panduan ini memakai ID ASLI (dari K1/K2) sehingga LULUS baik flag ON maupun OFF.

============================================================================
 LANGKAH 6 — K1 -> K2 : CLIENT BUAT PROJECT  (publish event)
============================================================================
 Method : POST
 URL    : {{BASE}}/bidding/projects
 Headers: Content-Type: application/json
          Authorization: Bearer {{clientToken}}     <- WAJIB (role client)
 Body   :
   { "judul_proyek": "Aplikasi Absensi Kampus",
     "deskripsi_proyek": "Bangun aplikasi absensi berbasis QR untuk fakultas",
     "skills": ["Backend"], "requirements": "Mahasiswa aktif, mampu Node.js",
     "budget_awal": 1000000, "kuota_maksimal": 1 }
 Harap  : 201 { "message": "Project created successfully", "data": {...} }
 SIMPAN : data.proyek_id -> {{projectId}}
 PENTING: "skills" HARUS dari enum K2 (Frontend, Backend, UI/UX, Mobile,
          Database, DevOps, Data Science, Machine Learning, ...).
 CATATAN: project.mitra_id otomatis = {{clientId}} (dari JWT).

============================================================================
 LANGKAH 7 — K2 -> K5 : VERIFIKASI EVENT PROJECT TER-CONSUME K5
============================================================================
 Method : GET
 URL    : {{BASE}}/notify/notifications
 Harap  : 200, memuat "routingKey":"bidding.project.created" &
          "project_title":"Aplikasi Absensi Kampus".

============================================================================
 LANGKAH 8 — K1 -> K2 : TALENT BID KE PROJECT
============================================================================
 Method : POST
 URL    : {{BASE}}/bidding/bids
 Headers: Content-Type: application/json
          Authorization: Bearer {{talentToken}}     <- WAJIB (role talent)
 Body (raw JSON) — student_id HARUS = {{talentId}} (id user K1 talent):
   { "project_id": {{projectId}},
     "student_id": "{{talentId}}",
     "priority": 1,
     "document_url": "https://github.com/nexus/proposal",
     "tawaran_harga": 950000,
     "tawaran_waktu": "2026-08-31" }
 Harap  : 201, body data bid
 SIMPAN : data.bid_id -> {{bidId}}
 KENAPA student_id = {{talentId}}: agar talent yg di-accept di K2 == talent yg
 dipakai di milestone K4 (rantai tertaut saat VALIDATE_INTEGRATION ON).

============================================================================
 LANGKAH 9 — K2 : CLIENT ACCEPT BID  -> talent "awarded" (publish event)
============================================================================
 Method : PUT
 URL    : {{BASE}}/bidding/bids/{{bidId}}/status
 Headers: Content-Type: application/json
          Authorization: Bearer {{clientToken}}     <- pemilik project
 Body   : { "status": "Accepted" }
 Harap  : 200, status_bid -> Accepted
 BUKTI INTEGRASI: K2 publish "bid_deal_confirmed" -> K5 (notif ke talent/client).
 EFEK: sekarang {{talentId}} terdaftar sbg accepted bidder di {{projectId}}.

============================================================================
 LANGKAH 10 — K4 : CLIENT BUAT MILESTONE (tertaut project + talent terpilih)
============================================================================
 Method : POST
 URL    : {{BASE}}/tracker/milestones
 Headers: Content-Type: application/json
 Body (raw JSON) — deadline WAJIB masa depan:
   { "title": "Laporan Akhir Sprint 1",
     "paymentAmount": 750000,
     "deadline": "2026-12-31T17:00:00Z",
     "projectId": "{{projectId}}",       <- TAUTAN ke project K2 (baru)
     "actorId": "{{clientId}}",          <- caller yg melakukan create milestone
     "employerId": "{{clientId}}",       <- client K1 (= pemilik project)
     "studentId": "{{talentId}}" }       <- talent K1 yg sudah di-accept
 Harap  : 201, body data.id + data.projectId terisi
 SIMPAN : data.id -> {{milestoneId}}
 K4 akan menolak (403/404/400) bila:
   - actorId != employerId, atau
   - employerId bukan client pemilik project, atau
   - project {{projectId}} tak ada di K2, atau
   - {{talentId}} belum di-ACCEPT di {{projectId}} (lihat Langkah 9).

============================================================================
 LANGKAH 12 — K4 : TALENT SUBMIT BUKTI (+file ke MinIO)
============================================================================
 Method : POST
 URL    : {{BASE}}/tracker/submissions
 Body (form-data, BUKAN JSON) — file HANYA PDF/JPG/PNG/WEBP (maks 10 MB):
     milestoneId | Text | {{milestoneId}}
     studentId   | Text | {{talentId}}     <- HARUS = studentId milestone
     actorId     | Text | {{talentId}}     <- caller submitter
     description | Text | Submission sprint 1
     file        | File | <pilih PDF/PNG/JPG/WEBP>   (boleh juga pakai "links")
 Harap  : 201, body data.id + data.fileUrl (URL MinIO)
 SIMPAN : data.id -> {{submissionId}} ; data.fileUrl -> {{fileUrl}}
 CEK    : {{fileUrl}} memuat "/documents/" (tersimpan di bucket MinIO).

============================================================================
 LANGKAH 14 — K4 -> K1 : CLIENT REVIEW SUBMISSION (+event K5)
============================================================================
 Method : POST
 URL    : {{BASE}}/tracker/submissions/{{submissionId}}/review
 Headers: Content-Type: application/json
 Body   — reviewerId HARUS = {{clientId}} (employer milestone-nya):
   { "reviewerId": "{{clientId}}", "status": "approved",
     "notes": "Pekerjaan sesuai, disetujui." }
 Harap  : 201, body memuat "...successfully"
 BUKTI INTEGRASI: K4 memanggil K1 (GET /internal/users/{{clientId}}) memastikan
 reviewer = client sah == employerId milestone. Lalu publish "submission_approved"
 -> K5. Status valid: approved | rejected | needs_revision.

============================================================================
 RINGKASAN URUTAN (cheat sheet)
============================================================================
  1. POST /auth/register     (client)                 -> 201
  2. POST /auth/login        (client)                  -> clientToken, clientId
  3. POST /auth/register     (talent)                  -> 201
  4. POST /auth/login        (talent)                  -> talentToken, talentId
  5. GET  /match/me          Bearer talent             -> role:student   [K1->K3]
  6. POST /bidding/projects  Bearer client             -> projectId      [K1->K2]
  7. GET  /notify/notifications                        -> cek project    [K2->K5]
  8. POST /bidding/bids      Bearer talent, student_id=talentId -> bidId  [K1->K2]
  9. PUT  /bidding/bids/:id/status Bearer client {Accepted} -> awarded    [K2->K5]
 10. POST /tracker/milestones projectId+employerId+studentId -> milestoneId [K4->K2]
 11. GET  /notify/notifications                        -> cek milestone  [K4->K5]
 12. POST /tracker/submissions  form-data + file       -> submissionId, fileUrl
 13. GET  {{fileUrl}} :9000                            -> baca berkas    [K4->MinIO]
 14. POST /tracker/submissions/:id/review reviewerId=clientId -> 201      [K4->K1->K5]

============================================================================
 TROUBLESHOOTING
============================================================================
 - 429 di /auth/login : rate-limit 10/15m. Pakai email lain (register bebas).
 - 400 "Validation failed" di /bidding/projects : "skills" di luar enum (L6).
 - 403 di /bidding/bids : token bukan role talent (L8). 403 di accept: bukan
   pemilik project (L9 harus pakai {{clientToken}} yg sama dgn L6).
 - 404/400 di /tracker/milestones saat VALIDATE_INTEGRATION ON :
     * "not found in the identity service" -> employerId/studentId bukan id K1.
     * "not found in the bidding service"  -> {{projectId}} salah.
     * "no accepted bid" -> talent belum di-accept (ulangi L8-L9 dgn id sama).
   (Saat flag OFF, validasi ini dilewati.)
 - 400 "Only PDF, JPG, PNG, or WEBP..." : field 'file' (L12) tipe tak diizinkan.
 - READ {{fileUrl}} gagal/timeout : port 9000 tak terjangkau (butuh tailnet)
   atau bucket policy public-read belum aktif.
 - 404 "Cannot GET /api/..." : path salah; pakai persis path di panduan ini.
